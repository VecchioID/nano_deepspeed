# 01 - LLM 训练挑战与并行策略概览

## 一、大模型训练的核心挑战

### 1.1 显存瓶颈

#### 通用显存公式

设模型参数量为 Ψ, 训练使用 Adam 优化器 + 混合精度 (BF16 前向/反向, FP32 主权重):

| 项目 | 符号 | 公式 | 说明 |
|------|------|------|------|
| 模型参数 (BF16) | P<sub>fp16</sub> | 2Ψ | 前向/反向使用的半精度权重 |
| 模型参数 (FP32) | P<sub>fp32</sub> | 4Ψ | Adam 维护的 32 位主权重 |
| 梯度 (BF16) | G | 2Ψ | 反向计算的梯度 |
| 优化器状态 (FP32) | O | 8Ψ | momentum (4Ψ) + variance (4Ψ) |
| **模型状态合计** | **M<sub>states</sub>** | **16Ψ** | P<sub>fp32</sub> + O + P<sub>fp16</sub> + G |
| **激活值** | **M<sub>act</sub>** | **≈ B·S·H·L·34 bytes** | 前向中间结果, 因配置而异 |

其中:
- **Ψ** = 模型参数量, **B** = batch size, **S** = 序列长度, **H** = hidden dim, **L** = 层数

#### 实例: LLaMA-65B

Ψ = 65 × 10⁹, 使用 Adam + BF16:

```
模型状态合计:    16Ψ = 16 × 65B × 1 byte = 1040 GB  ← 注意这是 bytes, 不是 bits
  └─ BF16 参数:   2Ψ = 130 GB
  └─ FP32 参数:   4Ψ = 260 GB
  └─ 梯度:        2Ψ = 130 GB
  └─ 优化器状态:  8Ψ = 520 GB

激活值 (B=1, S=4096, H=8192, L=80):
  ≈ B·S·H·(34L + 5) ≈ 1 × 4096 × 8192 × (2720 + 5) ≈ 91 GB

单卡 H100 (80GB) 纯显存需求: 1040 + 91 ≈ 1131 GB >> 80 GB
```

**关键结论**: 单卡放不下是大模型训练的首要现实。必须通过并行策略将模型状态和激活值分布到多卡。

### 1.2 计算瓶颈

- 单卡算力有限（H100: ~989 TFLOPS FP8）
- 计算-通信重叠需要精细调度
- 超长序列导致计算图巨大

### 1.3 通信瓶颈

- 跨节点带宽远小于单节点内带宽
  - NVLink: 900 GB/s (H100)
  - InfiniBand: 50 GB/s (NDR400)
  - 以太网: 25-100 GB/s
- All-reduce 等集合通信操作随卡数增加延迟上升

### 1.4 显存碎片与效率

- PyTorch CUDA allocator 会产生碎片
- 动态计算图导致内存峰值不可预测
- Activation 占用中间显存

  (Data Parallel)     (Model Parallel)    (Pipeline Parallel)

          │              ▼           ▼              │

### 2.1 数据并行 (Data Parallelism)

**核心思想**: 每个 GPU 持有完整模型副本，处理不同 batch 数据，通过 All-Reduce 同步梯度。

```
GPU 0: [模型副本] ← Batch 0 → 梯度 0 ─┐
GPU 1: [模型副本] ← Batch 1 → 梯度 1 ─┤
GPU 2: [模型副本] ← Batch 2 → 梯度 2 ─┤── [AllReduce] → 平均梯度 → 更新
GPU 3: [模型副本] ← Batch 3 → 梯度 3 ─┘
```

**DDP 通信量公式**:

设模型参数量 Ψ, 数据并行度 N, 每个参数 2 bytes (BF16):

| 操作 | 通信量 | 说明 |
|------|--------|------|
| All-Reduce (梯度) | 2Ψ (2 × N × Ψ bytes total) | reduce + broadcast |
| 每卡带宽开销 | 2Ψ × (N-1)/N | 实际每卡发送/接收的数据量 |

例子: Ψ = 7B, N = 64 → 每 step 通信 2 × 7 × 10⁹ = 14 GB

**优点**:
- 实现简单，计算利用率高
- 每个 GPU 独立执行前向/反向

**缺点**:
- 每卡需存储完整模型 → 显存墙
- All-Reduce 通信量与模型大小成正比

**适用场景**: 模型能塞进单卡，需要扩大 batch size

### 2.2 张量并行 / 模型并行 (Tensor Parallelism)

**核心思想**: 将单个算子（如 Linear）的矩阵乘法拆分到多个 GPU 上并行计算。

**Megatron-LM 方案 (1D 张量并行)**:

```
               ┌──────┐
               │ 输入  │
               └──┬───┘
                  │
          ┌───────┼───────┐
          ▼       ▼       ▼
      ┌──────┐┌──────┐┌──────┐
      │GPU 0 ││GPU 1 ││GPU 2 │  ← 每卡持有权重切片
      │W₁    ││W₂    ││W₃    │
      └──┬───┘└──┬───┘└──┬───┘
         │       │       │
         ▼       ▼       ▼
      ┌──────┐┌──────┐┌──────┐
      │Y₁=X·W₁││Y₂=X·W₂││Y₃=X·W₃│
      └──────┘└──────┘└──────┘
         │       │       │
         └───────┼───────┘
                 ▼
             All-Gather → Y = [Y₁, Y₂, Y₃]
```

**TP 通信量公式** (以列切分 Linear + All-Reduce 为例):

输入 X ∈ R<sup>B×S×H</sup>, 权重 W ∈ R<sup>H×H<sub>out</sub></sup>, TP 组大小 t

| 阶段 | 操作 | 每卡通信量 |
|------|------|-----------|
| 前向 | All-Reduce (各卡结果求和) | 2 × B × S × H<sub>out</sub>/t |
| 反向 (对输入梯度) | All-Reduce (各卡梯度求和) | 2 × B × S × H |
| 反向 (对权重梯度) | 无 (本卡持有权重列分片) | 0 |

**每层总通信**: 2BS(H + H<sub>out</sub>/t) bytes, 其中 B = batch, S = seq_len, H = hidden_dim

对 Transformer, H = H<sub>out</sub>, 每层通信 ≈ 2BSH(1 + 1/t) bytes

与 DDP 对比:
- DDP 每 step 通信 2Ψ bytes (整体一次)
- TP 每层通信 O(BSH), 总通信 O(L × BSH) = O(BS × 模型总参数)
- **TP 通信是逐层的, DDP 是整体的** → TP 通信可被计算掩盖, 但不跨节点

**优点**: 降低单卡显存，适合超大 hidden_dim
**缺点**: 每层都需要通信，扩展效率随卡数增加而下降

### 2.3 流水线并行 (Pipeline Parallelism)

**核心思想**: 将模型按 layer 切分，每张卡负责数个连续层，形成流水线。

**朴素流水线**:
```
GPU 0: Layer 1-4   →   GPU 1: Layer 5-8   →   GPU 2: Layer 9-12
```

**问题**: 存在大量气泡 (bubble)，GPU 利用率低。

#### 流水线气泡公式 (Bubble Ratio)

设流水线深度 p (卡数), micro-batch 数量 m (gradient accumulation steps):

**GPipe 调度**:

```
时间 →
GPU 0: F₀ F₁ F₂ F₃ | B₃ B₂ B₁ B₀
GPU 1:    F₀ F₁ F₂ F₃ | B₃ B₂ B₁ B₀
GPU 2:       F₀ F₁ F₂ F₃ | B₃ B₂ B₁ B₀

         ← 气泡 → ← 计算 → ← 气泡 →
```

```
气泡比例 (Bubble Ratio) = (p - 1) / (m + p - 1)
```

- p = 4, m = 8 → bubble = 3/11 ≈ 27%
- p = 4, m = 32 → bubble = 3/34 ≈ 8.8%
- **m 越大气泡越小**, 但总 batch size 增大, 收敛变慢

**1F1B (One-Forward-One-Backward) 调度** (PipeDream):

```
时间 →
GPU 0: F₀ F₁ F₂ B₂ B₁ B₀
GPU 1:    F₀ F₁ F₂ B₂ B₁ B₀
GPU 2:       F₀ F₁ F₂ B₂ B₁ B₀
```

- F: Forward, B: Backward, 下标为 micro-batch 编号
- 1F1B 交替前向/反向，气泡比例与 GPipe 相同, 但显存更低
- **更少的显存**是因为反向时立即释放 activation, 不需要保存所有 m 个 micro-batch 的中间结果

```
1F1B 气泡比例 ≈ (p - 1) / m    (m >> p 时)
```

**Interleaved 1F1B** (进一步降低气泡):
- 把每卡的 layer 再切分为更小的 chunk
- 气泡可降至 (p - 1) / (m × chunks + p - 1)

**优点**: 通信量小（仅传递 activation/gradient），适合跨机部署
**缺点**: 存在气泡，负载均衡需要精心设计

### 2.4 序列并行 (Sequence Parallelism)

**核心思想**: 在序列维度上切分长序列（如 32K/128K tokens）。

**做法**:
- 将输入序列切分到不同设备
- 注意力计算和 FFN 在设备间分布
- 关键通信: 每层需要一次 All-Gather/Reduce-Scatter

**应用**: DeepSpeed Ulysses, Ring Attention, Megatron-LM Sequence Parallel

## 三、混合并行 (3D Parallelism)

实际训练中通常组合使用**数据并行 + 张量并行 + 流水线并行** = **3D Parallelism**。

```
                     ┌──────────────────────┐
                     │       DP Group        │
                     │  (数据并行, 64卡)      │
                     └──────────────────────┘
                     │          │          │
              ┌──────┴──┐ ┌──────┴──┐ ┌──────┴──┐
              │ PP Group │ │ PP Group │ │ PP Group │
              │ (流水线)  │ │ (流水线)  │ │ (流水线)  │
              └──────┬───┘ └──────┬───┘ └──────┬───┘
              │  │  │      │  │  │      │  │  │
           ┌──┴──┴──┐  ┌──┴──┴──┐  ┌──┴──┴──┐
           │ TP GPUs│  │ TP GPUs│  │ TP GPUs│
           │(张量并行)│  │(张量并行)│  │(张量并行)│
           └────────┘  └────────┘  └────────┘
```

**典型配置** (以 64 卡训练 LLaMA-65B 为例):

| 维度 | 组大小 | 数量 | 说明 |
|------|--------|------|------|
| TP   | 8      | 8    | 节点内 8 卡张量并行 |
| PP   | 4      | 2    | 跨 4 个节点流水线 |
| DP   | 2      | 1    | 2 路数据并行 |

### 3.1 通信拓扑

- TP 组内: NVLink (节点内, 高速)
- PP 组内: NVLink/IB (节点内部/间)
- DP 组间: IB (跨节点, 较慢)

### 3.2 关键权衡

| 维度 | 优点 | 缺点 |
|------|------|------|
| TP 越大 | 单卡显存需求更低 | 通信开销大，扩展效率差 |
| PP 越大 | 通信少，跨机友好 | 气泡增加，GPU 利用率下降 |
| DP 越大 | 计算效率高 | 通信量随模型线性增长 |

**经验法则**:
- TP ≤ 节点内 GPU 数（通常 4-8）
- PP 深度 ≤ 4-8，避免过多气泡
- DP 作为全局扩展的主维度

## 四、并行策略定量对比

### 4.1 显存对比总表

设模型参数量 Ψ, 数据并行度 N, 使用 Adam 优化器 + BF16 混合精度:

| 策略 | 每卡显存 (模型状态) | 每 step 通信量 | 气泡 |
|------|-------------------|---------------|------|
| Naive DDP | 16Ψ | 2Ψ (All-Reduce) | 无 |
| ZeRO-1 | 4Ψ + 12Ψ/N | 2Ψ (All-Reduce) | 无 |
| ZeRO-2 | 2Ψ + 14Ψ/N | Ψ (Reduce-Scatter) | 无 |
| ZeRO-3 | 16Ψ/N | 3Ψ (2 All-Gather + Reduce-Scatter) | 无 |
| TP (t 路) | 16Ψ/t | 每层 O(BSH) | 无 |
| PP (p 段) | 16Ψ/p | 仅 activation 边界传递 | (p-1)/(m+p-1) |

### 4.2 通信效率对比

| 策略 | 通信模式 | 延迟敏感度 | 跨节点友好 | 扩展上限 |
|------|---------|-----------|-----------|---------|
| DP (All-Reduce) | 全局同步 | 低 (大消息) | 一般 | 千卡 |
| ZeRO-3 (All-Gather) | 逐层同步 | 中 (可重叠) | 好 | 千卡+ |
| TP (All-Reduce) | 逐算子同步 | 高 (小消息多) | 差 | 8卡 |
| PP (P2P) | 点对点 | 低 | 好 | 百卡 |

### 4.3 显存优化技术

#### 4.3.1 Activation Checkpointing (梯度检查点)

- **原理**: 前向时丢弃中间 activation，反向时重新计算
- **节省**: 显存从 O(L) 降到 O(√L) 或 O(1)
- **代价**: 约 20-30% 额外计算开销
- **粒度**: 按层 checkpoint 最常用

### 4.2 ZeRO (零冗余优化)

详见下一讲 [02-DeepSpeed ZeRO 优化详解](./02-DeepSpeed-ZeRO优化详解.md)

### 4.3 Mixed Precision Training

- FP16/BF16 前向+反向，FP32 主权重更新
- 显存减半，计算加速 2-4 倍
- BF16 优于 FP16（无溢出问题，动态范围更大）

### 4.4 Offloading

- CPU/NVMe offloading of optimizer states
- 训练超大模型时作为兜底策略
- 代价: CPU-GPU 传输带宽瓶颈

## 五、扩展定律 (Scaling Laws)

### 5.1 计算量公式 (FLOPs)

对 Transformer 模型, 一次训练 step (前向 + 反向) 的计算量:

```
总 FLOPs (前向+反向) ≈ 12 × Ψ × B × S
```

其中 Ψ = 参数量, B = batch size per step, S = 序列长度

推导依据:
- 前向 ≈ 6ΨT, 反向 ≈ 6ΨT (T = B × S, 总 token 数)
- 主要来自矩阵乘法 (Linear, Attention)

### 5.2 理想加速比

**Amdahl 定律** (固定问题规模, strong scaling):

```
加速比 Speedup(N) = 1 / ((1 - p) + p / N)
扩展效率 Efficiency(N) = Speedup(N) / N
```

| N | p=0.99 | p=0.999 |
|---|--------|---------|
| 8 | 7.5× (93%) | 7.9× (99%) |
| 64 | 39× (61%) | 59× (92%) |
| 256 | 80× (31%) | 204× (80%) |
| 1024 | 91× (9%) | 500× (49%) |

- p = 可并行化比例, N = 设备数
- **通信开销是主要的不可并行部分**, 因此 p 主要由计算/通信比决定

**Gustafson 定律** (随设备数扩大问题规模, weak scaling):

```
加速比 Speedup(N) = p + (1 - p) × N
```

- 大模型训练通常介于两者之间: 模型增大 + 数据增多

### 5.3 实际效率度量

**MFU (Model FLOPs Utilization)**:

```
MFU = 实际吞吐的 FLOPs / 理论峰值 FLOPs
    
     = (12 × Ψ × B × S / step_time) / (GPU数 × 单卡峰值FLOPs)
```

- H100 BF16 峰值: 989 TFLOPS
- 3D 并行 MFU 通常 40-55%
- 纯 DP + FlashAttention 可达 65-70%

**通信开销占比**:

```
通信开销占比 = 通信时间 / (计算时间 + 通信时间)
```

- TP 组内: 15-25% (NVLink, 高速)
- DP 组间: 5-15% (IB/RoCE, 可被计算掩盖)
- PP: <5% (只有 activation 传递)

### 5.4 Scaling 决策

| 因素 | 增加什么？ | 为什么？ |
|------|-----------|---------|
| 模型更大 | TP/PP | 单卡放不下 |
| 数据更多 | DP/PP | 提高吞吐 |
| 序列更长 | SP/TP | 注意力计算剧增 |
| 训练更快 | DP | 扩展效率最高 |
