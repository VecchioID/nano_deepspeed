# 04 - 框架对比与选型指南

## 一、主要框架总览

| 框架 | 开发者 | 核心能力 | 优势 | 劣势 |
|------|--------|---------|------|------|
| **DeepSpeed** | Microsoft | ZeRO-1/2/3, Offload, MoE | ZeRO 生态最成熟, 社区大 | TP/PP 支持较弱 |
| **FSDP** | PyTorch | ZeRO-3 类似功能 | 原生 PyTorch, 易用性好 | 高级特性少, 调优参数少 |
| **Megatron-LM** | NVIDIA | TP + PP + SP + DP | TP/PP 最成熟, MFU 高 | 学习曲线陡, 代码侵入性强 |
| **Megatron-DeepSpeed** | Microsoft+NVIDIA | Megatron TP/PP + DeepSpeed ZeRO | 组合最强, 业界主流 | 配置复杂 |
| **ColossalAI** | HPC-AI Tech | ZeRO + TP/PP + Sequence | 易用性好, 全流程覆盖 | 社区较小 |
| **Alpa** | UC Berkeley | 自动并行 | 自动规划并行策略 | 尚不成熟, 控制力弱 |
| **FairScale** | Meta | ZeRO/FSDP 早期实现 | 已被 FSDP 取代 | 不再维护 |

## 二、DeepSpeed vs FSDP 深度对比

### 2.1 设计理念

| 维度 | DeepSpeed | FSDP |
|------|-----------|------|
| 定位 | 独立训练框架 | PyTorch 原生组件 |
| 封装程度 | 引擎接管训练 loop | 模块化, 可组合 |
| 配置方式 | JSON 配置文件 | Python API |
| 定制性 | 黑盒 + 部分可配 | 白盒, 更易定制 |

### 2.2 功能对比

| 功能 | DeepSpeed ZeRO-3 | FSDP |
|------|------------------|------|
| 参数分片 | ✅ | ✅ |
| 优化器分片 | ✅ (ZeRO-1) | ❌ (仅 ZeRO-3 模式) |
| 梯度分片 | ✅ (ZeRO-2) | ✅ (内建) |
| CPU Offload | ✅ (Optimizer+Param) | ✅ (Optimizer+Param) |
| NVMe Offload | ✅ (ZeRO-Infinity) | ❌ |
| 通信重叠 | ✅ (精细控制) | ✅ (auto 模式) |
| 预取策略 | ✅ (3个参数可调) | ✅ (limit_all_gathers) |
| 混合精度 | ✅ (FP16/BF16) | ✅ (BF16 AMP) |
| Activation Checkpointing | ✅ | ✅ (内建) |
| MoE (Mixture of Experts) | ✅ | ❌ |
| 梯度累积 | ✅ | ✅ |
| 加载/保存 | 需要特殊处理 | HuggingFace 标准 |

### 2.3 性能对比 (LLaMA-13B, 64 A100)

| 指标 | DeepSpeed ZeRO-3 | FSDP |
|------|------------------|------|
| Max throughput | 185K tokens/sec | 178K tokens/sec |
| Peak memory | 42.3 GB | 44.1 GB |
| Communication overhead | 12.5% | 14.2% |
| MFU | 48.3% | 46.5% |
| Scaling efficiency (64→512) | 82% | 78% |

**结论**: DeepSpeed 在大多数场景下性能略优 (5-10%), 但差距在缩小。

### 2.4 代码对比

**FSDP 初始化**:
```python
from torch.distributed.fsdp import FullyShardedDataParallel as FSDP
from torch.distributed.fsdp import ShardingStrategy
from torch.distributed.fsdp.wrap import transformer_auto_wrap_policy

model = FSDP(
    model,
    sharding_strategy=ShardingStrategy.FULL_SHARD,
    auto_wrap_policy=transformer_auto_wrap_policy,
    cpu_offload=CPUOffload(offload_params=True),
    mixed_precision=MixedPrecision(
        param_dtype=torch.bfloat16,
        reduce_dtype=torch.bfloat16,
        buffer_dtype=torch.bfloat16,
    ),
    limit_all_gathers=True,
)
```

**DeepSpeed 初始化 (等价的配置)**:
```json
{
  "zero_optimization": {
    "stage": 3,
    "offload_param": {"device": "cpu"},
    "overlap_comm": true,
    "stage3_gather_16bit_weights_on_model_save": true
  },
  "bf16": {"enabled": true}
```

## 三、DeepSpeed vs Megatron-LM

### 3.1 核心定位差异

```
DeepSpeed:   数据并行优先, ZeRO 为核心
Megatron-LM: 模型并行优先, TP/PP 为核心

DeepSpeed 适合:  模型刚超出单卡 (7B-70B), GPU 数 8-256
Megatron-LM 适合: 超大模型 (70B+), GPU 数 256+
```

### 3.2 混合使用: Megatron-DeepSpeed

业界主流大模型训练方案: **Megatron 的 TP/PP + DeepSpeed 的 ZeRO**

```
┌─────────────────────────────────────────────┐
│            Megatron-DeepSpeed                │
├─────────────────────────────────────────────┤
│                                             │
│   张量并行 (TP) ← Megatron-LM              │
│       └─ 每个 Transformer 层内拆分          │
│                                             │
│   流水线并行 (PP) ← Megatron-LM            │
│       └─ 按 layer 切分                     │
│                                             │
│   数据并行 + ZeRO ← DeepSpeed              │
│       └─ TP/PP 组间的数据并行              │
│                                             │
│   Sequence Parallel ← Megatron-LM           │
│       └─ 长序列在注意力维度切分             │
└─────────────────────────────────────────────┘
```

**配置示例**:

```json
{
  "train_batch_size": 512,
  "tensor_model_parallel_size": 8,
  "pipeline_model_parallel_size": 4,

  "zero_optimization": {
    "stage": 1,
    "reduce_bucket_size": 5e8
  },

  "data_types": {
    "fp16": {"enabled": true, "loss_scale": 0}
  },

  "model": {
    "num_layers": 80,
    "hidden_size": 8192,
    "num_attention_heads": 64,
    "ffn_hidden_size": 28672,
    "seq_length": 4096
  }
}
```

### 3.3 为什么 Megatron-DeepSpeed 只用 ZeRO-1？

```
在 3D 并行中:
  TP 组内通信: All-Reduce per op (高频, 少量)
  PP 组内通信: P2P activation (低频, 大量)
  DP 组间通信: All-Reduce gradients (低频, 大量)

ZeRO-2/3 会与 TP 的 All-Reduce 产生通信冲突:
  - ZeRO-3: 每层 All-Gather(参数) + TP All-Reduce(计算) = 叠加
  - ZeRO-2: Reduce-Scatter(梯度) + TP All-Reduce(计算) = 叠加

ZeRO-1 只分片优化器状态, 不增加前向/反向通信,
与 TP/PP 的通信不会叠加冲突。
```

## 四、选型决策树

```
模型能塞进单卡 (如 1B)?
├─ 是 → DDP / ZeRO-1 (无需分片)
└─ 否 → 需要分布式训练

训练卡数 < 64?
├─ 是 → 纯 ZeRO-3 (DeepSpeed/FSDP)
│      ├─ 需要 HuggingFace 生态 → FSDP
│      └─ 需要高级功能 (Offload, MoE) → DeepSpeed
└─ 否 → 需要 3D 并行

训练卡数 64~512?
├─ 是 → DeepSpeed ZeRO-2/3 + 可选 TP=8
│      └─ 模型 ≤ 30B → ZeRO-3 alone
│      └─ 模型 > 30B → ZeRO-1 + TP=8 + PP=4~8
└─ 否 → 训练卡数 > 512, 模型 ≥ 70B
       └─ Megatron-DeepSpeed (标准 3D 并行)
       └─ TP=8 + PP=8~16 + DP=ZeRO-1

特殊需求:
├─ MoE 训练 → DeepSpeed (内建 MoE 支持)
├─ 超长序列 (32K+) → DeepSpeed Ulysses / Megatron SP
├─ 自动并行 → Alpa / FlexFlow
└─ 快速实验 → FSDP (最简配置)
```

## 五、不同规模的推荐方案

### 5.1 小规模实验 (1-4 卡, 模型 ≤ 3B)

```bash
# 方案: ZeRO-2, 不拆分 TP
deepspeed --num_gpus=4 train.py \
    --model_name meta-llama/Llama-2-7b-hf \
    --zero_stage 2 \
    --per_device_batch_size 4
```

### 5.2 中等规模 (8-32 卡, 7B-13B)

```bash
# 方案: ZeRO-3, 可选 TP=4
deepspeed --num_gpus=8 train.py \
    --zero_stage 3 \
    --tp_size 4 \
    --per_device_batch_size 2 \
    --grad_accum_steps 8 \
    --activation_checkpointing
```

### 5.3 大规模 (64-256 卡, 30B-70B)

```bash
# 方案: Megatron-DeepSpeed: TP=8 + PP=4~8 + ZeRO-1
deepspeed --num_gpus=8 --num_nodes=16 train.py \
    --tensor-model-parallel-size 8 \
    --pipeline-model-parallel-size 4 \
    --zero-stage 1 \
    --num-layers 80 \
    --hidden-size 8192 \
    --num-attention-heads 64 \
    --seq-length 4096
```

### 5.4 超大规模 (256+ 卡, 70B+)

```bash
# 方案: Megatron-DeepSpeed: TP=8 + PP=16 + ZeRO-1 + SP
# 结合 FlashAttention-3, Sequence Parallelism
deepspeed --num_gpus=8 --num_nodes=64 train.py \
    --tensor-model-parallel-size 8 \
    --pipeline-model-parallel-size 16 \
    --zero-stage 1 \
    --sequence-parallel \
    --flash-attention \
    --num-layers 80 \
    --hidden-size 8192 \
    --num-attention-heads 64 \
    --seq-length 32768
```

## 六、混合精度方案

| 精度 | 位宽 | 动态范围 | 精度 | DeepSpeed 配置 | 推荐场景 |
|------|------|---------|------|----------------|---------|
| FP32 | 32-bit | ~84dB | 高 | 不推荐 | 优化器主权重 |
| FP16 | 16-bit | ~30dB | 中 | `fp16: {enabled: true}` | 旧硬件(V100) |
| BF16 | 16-bit | ~62dB | 低 | `bf16: {enabled: true}` | A100/H100 首选 |
| FP8 (E4M3) | 8-bit | ~18dB | 低 | `fp8: {enabled: true}` | H100 实验性 |

**推荐**: BF16 前向+反向, FP32 主权重, ZeRO 分片优化器状态。

## 七、未来趋势

### 7.1 FlashAttention 系列

- FlashAttention-1/2: 加速 Attention, 减少显存
- FlashAttention-3: FP8 支持, 更低延迟
- Ring Attention: 分布式长序列处理

### 7.2 稀疏化与 MoE

- Mixtral 8x7B: 推理时只激活 2/8 expert
- DeepSpeed-MoE: 支持 MoE 训练的 ZeRO 优化
- 未来: 稀疏化是扩展模型规模的关键路径

### 7.3 FP8 训练

- H100 原生支持 FP8
- 计算吞吐: FP8 = 2× BF16
- 挑战: 精度损失, loss scale 管理
- DeepSpeed FP8: 实验性支持

### 7.4 训练-推理统一框架

- vLLM + DeepSpeed 集成
- TensorRT-LLM
- 一个框架同时处理训练和推理
